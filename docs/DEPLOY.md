# Production tractusx-edc per-participant deployment (Alice + Bob)

Replaces the broken `-memory` charts (isolated in-memory vaults) with the full
charts backed by a **single HashiCorp Vault shared per participant** between that
participant's IdentityHub and connector. The IH-generated STS client secret and DID
signing key therefore land in the same vault the connector reads — no manual bridge.

Central services (`auth-windx.cluster.swms-cloud.com`: issuer / BDRS / did:web / DCP
issuance / status list) are **untouched**.

```
namespace windx-alice                       namespace windx-bob
 ┌───────────────────────────────┐           ┌───────────────────────────────┐
 │ alice-vault  (shared)         │           │ bob-vault   (shared)          │
 │ alice-postgres (db: ih, edc)  │           │ bob-postgres (db: ih, edc)    │
 │ alice-ih   (tractusx-identityhub 0.3.2)   │ bob-ih    (tractusx-identityhub 0.3.2)
 │ alice-edc  (tractusx-connector 0.12.1)    │ bob-edc   (tractusx-connector 0.12.1)
 │ bob-backend (nginx) — keep    │           │                               │
 └───────────────────────────────┘           └───────────────────────────────┘
```

| Secret                        | Value used here                                     | Notes |

## Chart value key-paths that make the shared-vault STS wiring work

IdentityHub (`ih-full-<p>.yaml`) → ConfigMap `*-config` / `*-datasource-config`:
- `vault.hashicorp.url` → `edc.vault.hashicorp.url` = `http://<p>-vault:8200`
- `vault.hashicorp.token` → `edc.vault.hashicorp.token` = `root`
- `postgresql.jdbcUrl` / `postgresql.auth.*` → `edc.datasource.*` (db `ih`)
- `identityhub.didweb.https: true` → `edc.iam.did.web.use.https=true`
- `identityhub.endpoints.identity.authKeyAlias: sup3r$3cr3t` → `web.http.identity.auth.alias` (**vault alias**, not a literal key)
- `identityhub.env.EDC_IAM_STS_PRIVATEKEY_ALIAS` / `EDC_IAM_STS_PUBLICKEY_ID` = `<did>#signing-key-1`
- `identityhub.iatp.sts.oauth.client.enabled: false` → `edc.tractusx.ih.participant.configurable.enable=false` (we provision via the Identity API instead)

Connector (`conn-full-<p>.yaml`) → controlplane/dataplane deployment env:
- `iatp.id` → `EDC_PARTICIPANT_ID`, `EDC_IAM_ISSUER_ID`
- `participant.id` → `TRACTUSX_EDC_PARTICIPANT_BPN`; `participant.contextId` → `EDC_PARTICIPANT_CONTEXT_ID`
- `iatp.sts.oauth.token_url` → `EDC_IAM_STS_OAUTH_TOKEN_URL` = `http://<p>-ih:8087/api/sts/token`
- `iatp.sts.oauth.client.id` → `EDC_IAM_STS_OAUTH_CLIENT_ID`
- `iatp.sts.oauth.client.secret_alias` → `EDC_IAM_STS_OAUTH_CLIENT_SECRET_ALIAS` = **`<did>-sts-client-secret`** ← same alias the IH writes
- `iatp.trustedIssuers[0]` → `EDC_IAM_TRUSTED-ISSUER_0-ISSUER_ID` / `_SUPPORTEDTYPES`
- `controlplane.bdrs.server.url` → `TX_EDC_IAM_IATP_BDRS_SERVER_URL` = `https://auth-windx.cluster.swms-cloud.com/api/directory`
- `dataplane.token.signer.privatekey_alias` → `EDC_TRANSFER_PROXY_TOKEN_SIGNER_PRIVATEKEY_ALIAS` = `<did>#signing-key-1`
- `dataplane.token.verifier.publickey_alias` → `EDC_TRANSFER_PROXY_TOKEN_VERIFIER_PUBLICKEY_ALIAS` = `<did>#signing-key-1`
- `vault.hashicorp.*` / `postgresql.*` → same shared vault + db `edc`

## Ordered deploy sequence

Run the whole thing once per participant. Below is **Alice**; repeat with the `-bob`
files, namespace `windx-bob`, DID/host/BPN swapped.

### 0. Prereqs
```bash
helm repo add tractusx-edc https://eclipse-tractusx.github.io/charts/dev   # already added
helm repo update
export KUBECONFIG=...           # cluster with nginx ingress + cert-manager (letsencrypt-prod)
```

### 1. Remove the old `-memory` releases (keep bob-backend)
```bash
helm -n windx-alice uninstall alice-ih  || true
helm -n windx-alice uninstall alice-edc || true
helm -n windx-bob   uninstall bob-ih    || true
helm -n windx-bob   uninstall bob-edc   || true
# bob-backend nginx stays.
```

### 2. Shared Vault + Postgres (namespaces already exist)
```bash
kubectl -n windx-alice apply -f vault-alice.yaml
kubectl -n windx-alice apply -f postgres-alice.yaml
kubectl -n windx-alice rollout status deploy/alice-vault
kubectl -n windx-alice rollout status deploy/alice-postgres
```

### 3. Seed the IH Identity-API super-user key into the shared vault
The full chart authenticates the Identity API with the key stored in the vault at
`identityhub.endpoints.identity.authKeyAlias` (`sup3r$3cr3t`) — **not** a literal env var.
Seed it with the MXD default so `X-Api-Key: c3VwZXItdXNlcg==.c3VwZXItc2VjcmV0LWtleQo=` works:
```bash
kubectl -n windx-alice exec deploy/alice-vault -- \
  sh -c 'VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root \
         vault kv put secret/sup3r\$3cr3t content="c3VwZXItdXNlcg==.c3VwZXItc2VjcmV0LWtleQo="'
```
> The EDC hashicorp-vault extension reads the value from the `content` field (MXD convention).

### 4. IdentityHub (full chart)
```bash
helm upgrade --install alice-ih tractusx-edc/tractusx-identityhub \
  --version 0.3.2 -n windx-alice -f ih-full-alice.yaml
kubectl -n windx-alice rollout status deploy/alice-ih
```
Confirm did:web is public + HTTPS (cert-manager may take a minute):
```bash
curl -s https://alice-windx.cluster.swms-cloud.com/.well-known/did.json | jq .
```

### 5. Provision the participant context + STS account (super-user Identity API)
This is the MXD `SeedIH → Create Participant` call, adapted to our DIDs/hosts. It:
creates the participant context, **generates the DID signing key** (EdDSA/Ed25519) into the
shared vault at `<did>#signing-key-1`, **creates the STS account** and stores its client
secret in the shared vault at `<did>-sts-client-secret`, and sets the DID-doc service
endpoints (CredentialService + ProtocolEndpoint). **We OMIT credential seeding** — the
MembershipCredential is issued later via our own central DCP flow.

```bash
# reach the internal Identity API (port 8082) via port-forward
kubectl -n windx-alice port-forward svc/alice-ih 8082:8082 &

curl -s -X POST http://localhost:8082/api/identity/v1alpha/participants \
  -H 'Content-Type: application/json' \
  -H 'X-Api-Key: c3VwZXItdXNlcg==.c3VwZXItc2VjcmV0LWtleQo=' \
  -d '{
    "active": true,
    "did": "did:web:alice-windx.cluster.swms-cloud.com",
    "key": {
      "keyGeneratorParams": { "algorithm": "EdDSA", "curve": "Ed25519" },
      "keyId":           "did:web:alice-windx.cluster.swms-cloud.com#signing-key-1",
      "privateKeyAlias": "did:web:alice-windx.cluster.swms-cloud.com#signing-key-1"
    },
    "participantId": "did:web:alice-windx.cluster.swms-cloud.com",
    "serviceEndpoints": [
      {
        "type": "CredentialService",
        "serviceEndpoint": "https://alice-windx.cluster.swms-cloud.com/api/credentials/v1/participants/ZGlkOndlYjphbGljZS13aW5keC5jbHVzdGVyLnN3bXMtY2xvdWQuY29t",
        "id": "credentialservice-1"
      },
      {
        "type": "ProtocolEndpoint",
        "serviceEndpoint": "https://alice-edc-windx.cluster.swms-cloud.com/api/v1/dsp",
        "id": "dsp-url"
      }
    ]
  }'
```
Bob equivalents: DID `did:web:bob-windx.cluster.swms-cloud.com`, base64 segment
`ZGlkOndlYjpib2Itd2luZHguY2x1c3Rlci5zd21zLWNsb3VkLmNvbQ==`, CredentialService host
`bob-windx...`, ProtocolEndpoint `https://bob-edc-windx.cluster.swms-cloud.com/api/v1/dsp`.

> NOTE the endpoint paths vs MXD: connector DSP path in chart 0.12.1 is **`/api/v1/dsp`**
> (MXD used `/api/dsp`), and did:web is **HTTPS**.

### 6. Confirm the STS secret + DID key are in the shared vault
```bash
kubectl -n windx-alice exec deploy/alice-vault -- \
  sh -c 'VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root \
    vault kv get "secret/did:web:alice-windx.cluster.swms-cloud.com-sts-client-secret"'
kubectl -n windx-alice exec deploy/alice-vault -- \
  sh -c 'VAULT_ADDR=http://127.0.0.1:8200 VAULT_TOKEN=root \
    vault kv get "secret/did:web:alice-windx.cluster.swms-cloud.com#signing-key-1"'
```
Both must exist **before** the connector starts (it reads them at boot).

### 7. Connector (full chart)
```bash
helm upgrade --install alice-edc tractusx-edc/tractusx-connector \
  --version 0.12.1 -n windx-alice -f conn-full-alice.yaml
kubectl -n windx-alice rollout status deploy/alice-edc-controlplane
kubectl -n windx-alice rollout status deploy/alice-edc-dataplane
```
Smoke test — the connector should mint an STS token against its IH (no vault errors in log):
```bash
kubectl -n windx-alice logs deploy/alice-edc-controlplane | grep -i -E 'sts|vault|error' | tail
```

### 8. Repeat 2–7 for Bob (windx-bob, `*-bob.yaml`).

## Alternative provisioning: chart-native "configurable initial participant" (no POST)
If you prefer to avoid the Identity-API POST, set in `ih-full-<p>.yaml`:
```yaml
identityhub:
  iatp:
    sts:
      oauth:
        client:
          enabled: true
          id: "did:web:alice-windx.cluster.swms-cloud.com"
          secret: "<a secret YOU choose>"
          secret_alias: "did:web:alice-windx.cluster.swms-cloud.com-sts-client-secret"
          x_api_key: "ZGlkOndlYjphbGljZS13aW5keC5jbHVzdGVyLnN3bXMtY2xvdWQuY29t.<random>"
```
→ maps to `edc.tractusx.ih.participant.configurable.*`. The chart then creates the context
and stores the STS secret **you chose** at the alias **you chose** (deterministic; solves the
"capture the generated secret" problem directly). Trade-off: it does **not** let you set the
DID-doc `CredentialService`/`ProtocolEndpoint` service endpoints or `keyGeneratorParams`, and
the DID-signing-key alias it uses is not documented — so for data-plane key reuse you'd then
provision a separate signer keypair into the vault yourself. Pick ONE of the two approaches,
not both (a second create would conflict).

## Does the full IH's super-user Identity API auth work? (MXD relies on it)
**Expected: yes**, but the mechanism differs from MXD. MXD's custom image took the literal
`EDC_IH_API_SUPERUSER_KEY`. The 0.3.2 chart instead exposes
`identityhub.endpoints.identity.authKeyAlias` → `web.http.identity.auth.alias`, which is a
**vault alias**; the runtime reads the actual key from the vault. So you must **seed the vault**
(step 3) with the key you intend to send as `X-Api-Key`. Using the MXD default value keeps the
header identical to MXD. `ih-full-*.yaml` also injects `EDC_IH_API_SUPERUSER_KEY` as a
belt-and-suspenders literal, but the authoritative path is the vault alias.

## Remaining unknowns to verify live
1. **Whether the `tractusx/identityhub:0.3.2` image honours `EDC_IH_API_SUPERUSER_KEY` as a
   literal**, or ONLY the vault `web.http.identity.auth.alias`. Seed the vault regardless
   (step 3). If POST returns 401, the literal env is ignored — the vault seed is the fix.
2. **STS client-secret alias convention** — that provisioning stores the secret at exactly
   `<participantContextId>-sts-client-secret`. This is the MXD assumption; verify with step 6
   before deploying the connector. If the alias differs, read the real one and set
   `iatp.sts.oauth.client.secret_alias` in the connector values to match.
3. **`participant.contextId`** — set to the DID here (single-tenant). Confirm the connector
   accepts the DID as `EDC_PARTICIPANT_CONTEXT_ID`; if a UUID is required, generate one.
4. **DNS** — `alice-edc-windx` / `bob-edc-windx.cluster.swms-cloud.com` must resolve like the
   existing `*-windx` hosts (assumed wildcard). Confirm before relying on the DSP/data ingress
   + LE cert issuance.
5. **did:web document path** — verify the IH serves the DID doc at
   `https://<host>/.well-known/did.json` for a host-only DID (`did` endpoint path `/`).
   Confirm with the curl in step 4.
6. **EDC hashicorp-vault KV field** — assumes the extension reads the `content` field on a
   dev-mode KV v2 mount at `secret/` with `paths.secret=/v1/secret` (MXD-proven). If vault
   reads fail, check whether the image expects KV v1 or a different field.
7. **STS token-signing key** — `EDC_IAM_STS_PRIVATEKEY_ALIAS`/`_PUBLICKEY_ID` are set to
   `<did>#signing-key-1` (the participant DID key). Confirm the STS actually signs with the
   per-participant key and that this global env doesn't shadow it; MXD used a separate default
   `key-1`. If STS token issuance fails, fall back to `key-1` and seed that key in the vault.
